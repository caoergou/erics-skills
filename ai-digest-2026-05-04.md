### 1. AWS MCP Server is Now Generally Available
> AWS 在 5 月 6 日正式 GA 自家的 MCP Server，提供受 IAM、CloudWatch、CloudTrail 全面治理的 AWS 资源访问通道，给 AI 编码 Agent 一条"安全可审计"的进 AWS 后门。

- 📰 来源：AWS What's New · [原文](https://aws.amazon.com/about-aws/whats-new/2026/05/aws-mcp-server/)
- 🏷 分类：Agent
- 💬 深度点评：
  **背景与意义**：MCP 从 2024 年底由 Anthropic 抛出至今，最大悬念一直是"超大云厂商会不会下场"。AWS 的 GA 等于给 MCP 盖上了 enterprise 级的合规章——这意味着 MCP 不再只是开发者插件协议，而是即将被写进甲方采购合同的标准。
  **谁该关注**：1) 在 AWS 上跑业务的平台/SRE 团队；2) 想把 Claude Code、Cursor 接进生产环境的研发负责人；3) 做 Agent 编排框架的同行——你的"自研 AWS 适配"很可能本周就过气了。
  **对比与定位**：相对早些时候 Cloudflare、GitHub 推出的 MCP Server，AWS 这版的杀手锏是"原生 IAM + CloudTrail 审计"。Jama Software 5 月 4 日的 MCP Server（针对工程管理流程）属另一象限——AWS 占的是基础设施层。
  **硬伤与风险**：1) 服务范围还在"managed server"阶段，许多 niche 服务没覆盖；2) IAM 颗粒度是把双刃剑——配错权限比配 SSH key 更容易酿事故；3) 计费模式尚不透明，Agent 调用频次极高时账单会失控。
  **趋势判断**：早期趋势确认。本月内 Azure / GCP 必跟进；建议在 staging 环境先接入，权限配置走最小必要原则，并把 CloudTrail 日志接入 SIEM。

  👍 值得点：原生 IAM 治理 + CloudTrail 审计 + CloudWatch 指标三件套齐全，是目前合规度最高的 MCP 实现
  👎 需注意：AWS 的 managed server 通常意味着区域受限和价格"灵活"，国内/边缘场景慎入

### 2. GPT-5.5 / GPT-5.5 Pro Released, Reclaiming Top Spot on ARC-AGI-2
> OpenAI 4 月 24 日上线 GPT-5.5，ARC-AGI-2 (Verified) 跳到 85.0%（Opus 4.7 75.8%、Gemini 3.1 Pro 77.1%），Terminal-Bench 2.0 82.7%；本周三大模型对比开始在社区刷屏。

- 📰 来源：OpenAI / Artificial Analysis · [原文](https://openai.com/index/introducing-gpt-5-5/) · [独立分析](https://artificialanalysis.ai/articles/openai-gpt5-5-is-the-new-leading-AI-model)
- 🏷 分类：Foundation Model
- 💬 深度点评：
  **背景与意义**：自 4.6 → 4.7 以来 Anthropic 一直握住 SWE/Agent 任务的口碑高地，GPT-5.5 这轮把 ARC-AGI-2、Terminal-Bench、APEX-Agents 一次性拿走，标志着 OpenAI 暂时夺回"通用 frontier"位。但本期更值得记的，是 OpenAI 把 GPT-5.5 Pro 推上 API 同步发布——意味着竞争已从"模型发布会秀肌肉"转向"开发者价格 × 性能 × 上下文窗口"的硬指标对决。
  **谁该关注**：1) 跑 Agent harness 的产品团队，要重新评估 default model；2) 做评测基准的研究者，ARC-AGI-2 跳分意味着旧基准可能再次饱和；3) 投放 token 预算的 CTO——别再盯着 4.7 vs 5.5 的小数点。
  **对比与定位**：在"长程编程任务"上 Claude Opus 4.7 仍有口碑优势（5 月初社区反馈 Opus 在数小时连续 refactor 任务上输出更稳）；在"推理 + 工具调用"上 GPT-5.5 (xhigh) 更强；Gemini 3.1 Pro 的差异化点是多模态视觉与工作流集成。
  **硬伤与风险**：1) ARC-AGI-2 系基准在数月内连跳两个台阶，存在 overfitting / 数据污染嫌疑；2) xhigh 推理模式 token 消耗与延迟未公开，实际成本可能远超普通 high；3) 同质化加剧——三家在 +/-3pp 区间互相超越，对开发者意味着"换模型不再是 productivity 大跳变"。
  **趋势判断**：成熟期信号——头部模型差距进入"两位数小数"。建议构建"模型路由"层（按任务类型分流），而非锁死单一供应商。

  👍 值得点：ARC-AGI-2 +11.7pp 的跳幅是过去半年单次升级最大；API 同步开放
  👎 需注意：xhigh 模式实际定价与限流未明确披露，盲目切换会被账单教训

### 3. Addy Osmani Launches `agent-skills` — Production-Grade Skills for AI Coding Agents
> Google Chrome 高级工程师 Addy Osmani 发布 19 个"高级工程师必做但 Agent 总跳过"的 skill，组织成 6 个生命周期阶段 + 7 个 slash command；GitHub 本周 trending 持续上榜。

- 📰 来源：addyosmani.com · [博客](https://addyosmani.com/blog/agent-skills/) · [GitHub](https://github.com/addyosmani/agent-skills)
- 🏷 分类：Dev Tool / Methodology
- 💬 深度点评：
  **背景与意义**：行业终于承认"模型变强 ≠ 工程变好"。Agent 像"零经验但手快的初级工程师"——会写 diff，但不会写 spec、不留 evidence、不评估变更范围。Addy 的 agent-skills 把"senior engineering 的隐性知识"显性化、自动化、不可跳过——这是从"提示词工程"向"流程工程"的一次清晰位移。
  **谁该关注**：1) 团队规模 5+ 人、被 AI 生成代码淹没的 lead/staff engineer；2) 在做 AI coding 内训规范的工程效能团队；3) 任何在 Cursor / Claude Code / Codex 上跑生产代码的人。
  **对比与定位**：与 obra/superpowers（Jesse Vincent，Prime Radiant）形成"姐妹项目"——superpowers 重 brainstorming/TDD/debug 等通用思维 skill，agent-skills 重 spec→plan→build→test→review→ship 工程闭环。两者可叠加，并不互斥。相比 .cursorrules / CLAUDE.md 这种轻量约定，它强制"evidence-required"门禁。
  **硬伤与风险**：1) 19 个 skill + 7 个 slash command，对独立开发者属于过度工程；2) 强制"留 evidence"会让 PR 描述膨胀，code review 时间不一定降；3) 跨工具兼容靠"约定"，Cursor / Windsurf 等不同 harness 上效果差异未量化。
  **趋势判断**：早期趋势但拐点已到——"Agent 工程方法论"会在 2026 下半年成为面试题与团队规范的一部分。建议先在内部一个 service 上跑 spec-driven-development 这一个 skill，再决定全套。

  👍 值得点：每个 skill 都列出"agent 常用的偷懒借口 + 反驳"，是黑色幽默式实用主义
  👎 需注意：与团队既有 PR 模板/CI 流水冲突时，迁移成本可能高于预期

### 4. DeepSeek V4 与 Kimi K2.6：开源权重模型贴近闭源前沿
> DeepSeek 4 月 24 日发布 V4（Pro 1.6T MoE / 49B active；Flash 284B / 13B；1M 上下文，开源权重），Kimi K2.6（4 月 20 日，1T VLM）成为首个在 SWE-Bench Pro 击败 GPT-5.4 (xhigh) 的开源权重模型。

- 📰 来源：DeepLearning.AI The Batch · [原文](https://www.deeplearning.ai/the-batch/kimi-k2-6-matches-open-qwen3-6-max-anddeepseek-v4-falls-just-behind-top-closed-models) · [Fortune DeepSeek V4](https://fortune.com/2026/04/24/deepseek-v4-ai-model-price-performance-china-open-source/)
- 🏷 分类：Foundation Model / Open Source
- 💬 深度点评：
  **背景与意义**：开源权重和闭源前沿的差距从过去的 6-12 个月，压缩到现在的 2-4 周。本周社区 benchmark 帖子开始默认把 DeepSeek V4 / Kimi K2.6 与 Opus 4.7 / GPT-5.5 放进同一张表——这件事本身比单一分数更值得记录。
  **谁该关注**：1) 关心数据主权的 enterprise；2) 想做 fine-tune / 蒸馏的研究者；3) 国内做 Agent 应用、对 token 单价敏感的团队。
  **对比与定位**：DeepSeek V4 走"低价 + 与华为芯片深度集成"路线，Kimi K2.6 是 1T VLM（视觉强），Qwen3.6-35B-A3B 走"小活跃参数"高效路线。三者覆盖了"超大上下文 / 多模态 / 推理高效"三种需求位。
  **硬伤与风险**：1) MoE 部署对显存与 KV cache 要求极高，自托管成本未必比 API 低；2) SWE-Bench Pro 这种基准对模型口味敏感，单次跑分不能等同生产可用；3) 开源权重 ≠ 开源数据 / 开源训练流程，复现仍困难。
  **趋势判断**：成熟期信号——"是否上开源权重"会成为 enterprise 选型的硬约束。建议在评测时把"自托管 TCO"列入基准，而非只看 quality。

  👍 值得点：1M 上下文窗口 + 开源权重 + permissive license 的组合在 4 月之前没人做到
  👎 需注意：MoE 推理部署比想象中贵，"开源 = 便宜"是错觉

> Anthropic 本月放出 Project Glasswing 内测，配合 Claude Mythos Preview 在数周内识别"数千个零日漏洞"，包括一个 27 岁的 OpenBSD bug，引发安全圈与商业圈双重震动。

- 📰 来源：Fortune / IMFounder · [Fortune 报道](https://fortune.com/2026/05/02/agentic-ai-governance-framework-banking-healthcare-retail-supply-chain-yale-celi-sonnenfeld/) · [汇总](https://imfounder.com/science-tech/ai/ai-updates-may-2026/)
- 🏷 分类：Industry Viewpoint / Security
- 💬 深度点评：
  **背景与意义**：这是"Agent 走出 demo 走进生产"的标志事件之一。AI 找漏洞过去通常被讥为"刷 CVE 数量"，但能在 OpenBSD（业内公认审计最严的代码库之一）里挖到 27 年陈年问题——意味着 LLM 已能进行人脑难以维持的"超长上下文交叉对照"。
  **谁该关注**：1) 安全团队 / red team；2) 关键基础设施维护者（kernel / TLS / libc 上下游）；3) 政策制定者——攻防失衡论调会再次抬头。
  **对比与定位**：相比 Google Project Naptime / GitHub Copilot Autofix，Glasswing 的"内测筛选 + 责任披露"路径更克制，但商业化暗示明确。Mythos 是 Anthropic 暗示"比 Opus 4.7 更强"但暂未发布的模型——本周社区里"Mythos 何时公开"成为热搜词。
  **硬伤与风险**：1) "数千个零日"的口径未公开 CVE 编号，无法独立核验；2) 找到 ≠ 修复，下游 patch / 补丁分发链尚未形成；3) 攻防对称问题——这种能力在恶意手中等于灾难。
  **趋势判断**：早期趋势但战略意义巨大。本年内会出现"AI vs AI 漏洞挖掘"对抗格局。建议安全团队启动 AI-assisted code review 试点，但所有 finding 必须人工二次验证。

  👍 值得点：把"超长上下文"用在了真正能利用其优势的场景（漏洞挖掘 = 跨文件交叉引用）
  👎 需注意：营销口径"thousands of zero-days"目前缺独立审计，对外引用要谨慎

## 💡 重要观点 & 深度分析

### Long-Horizon Agents Are Here. Full Autopilot Isn't.
> 5 小时连续自主工作 + 任务时长每 196 天翻倍——但社区开始警告"长程 Agent 大多是脆弱的循环，只是还没失败而已"。

深度点评：
**背景与意义**：当所有厂商都在拼"任务时长"时，这篇文章把"任务时长翻倍 ≠ 任务可靠"这个不舒服的事实摆上桌。它和 Melbourne 那篇研究形成对照：一个说"Agent 越跑越久"，一个说"编排越多越糟"——共同指向"我们高估了 Agent 的鲁棒性"。
**谁该关注**：1) 把 Agent 做进核心业务流的产品经理；2) 想用 Agent 替代人工 ops 的运营负责人；3) 评估 Agent 项目 ROI 的财务/采购。
**对比与定位**：与 Sequoia "2026: This is AGI" 的乐观叙事正面冲突。这种 demo vs prod 之间的"可靠性鸿沟"和早期自动驾驶 L2-L4 的撕裂极为相似。
**硬伤与风险**：观点本身可能过度悲观——确实有窄域 Agent（如安全审计）已经稳定产出价值。把所有"长程 Agent"一锅端否定也是错。
**趋势判断**：本季度内会出现新的"Agent 可靠性"基准（不是任务长度，而是任务连续成功率）。建议在生产部署前设定 SLO，并默认接入 human-in-the-loop。

👍 值得点：直面"漂移"问题，给"context rot"现象一个准确命名
👎 需注意：把所有长程 Agent 都贴上"脆弱循环"标签略草率，安全/审计等场景反例存在

### Do We Still Need LangGraph?
> Melbourne 大学研究显示"在大多数程序性任务上，外部 Agent 编排框架让 AI 表现更差"——LangGraph / CrewAI / AutoGen 的存在合理性首次被严肃挑战。

深度点评：
**背景与意义**：过去两年 Agent 圈默认"模型 + 编排框架 = 生产解"，这次研究把 baseline（直接 prompt）拉回桌面对比，结论是"很多场景不如直给"。本周 X 上 LangChain 与 CrewAI 维护者已亲自下场回应。
**谁该关注**：1) 已经在生产用 LangGraph/CrewAI 的团队；2) 准备引入编排框架的架构师；3) 做 Agent 评测的研究者。

**趋势判断**：行业会从"all-in 框架"回摆到"按需引入"。建议先用 plain SDK 做 MVP，确认有真实编排痛点（多 Agent / 状态机 / 人审）再上框架。
